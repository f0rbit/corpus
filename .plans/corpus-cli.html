Plan: corpus CLI

todo

A read-only corpus CLI (list stores, list versions, show metadata + observations, cat decoded data, lineage) plus clone (prod→file, file→file), built on two new library-core capabilities: a laptop-side remote backend (Cloudflare D1 HTTP + R2 S3 API implementing the existing Backend surface) and a generic copy(source, dest) with content-hash-skip incremental semantics. The CLI is a thin veneer shipped as a bin inside @f0rbit/corpus — zero new runtime dependencies (node:util parseArgs, Bun.S3Client, native fetch). Config discovery follows the drizzle-kit pattern: an optional corpus.config.ts in the consumer workspace supplies store definitions (required only for cat decoding), while wrangler.toml/wrangler.jsonc sniffing + env vars cover connection params so metadata commands work with NO config. Targets the unpublished 0.8.0.

Root cause

Stated goal: A corpus CLI that discovers D1/R2 config from the workspace, inspects file-system backends, lists/views snapshot metadata, and clones prod into a local file backend.

Underlying need: Operational visibility into versioned snapshot data across environments — today every "what's actually in prod?" question means a throwaway script or the Cloudflare dashboard, and there is no sanctioned way to reproduce prod data locally for debugging. The missing primitives are library-level: corpus has no backend reachable from a laptop (the Cloudflare backend needs Worker bindings) and no cross-backend copy operation.

Why this matters: Solving this as "just a CLI" (ad-hoc REST calls + a clone loop inside the CLI) would duplicate the SQL/row-mapping layer that backend/cloudflare.ts already owns, and lock incremental copy — useful to every corpus consumer, including the pipelines orchestrator — inside one tool. Putting the remote backend and copy() in library core makes the CLI a veneer and the capabilities reusable, contract-tested, and property-tested like every other backend.

Alternatives considered

User-fixed constraints (not re-litigated): prod access is direct Cloudflare APIs (D1 HTTP REST + R2 S3-compatible), and v1 is read-only + clone. The axes explored below are the remaining open ones: remote-backend transport shape, packaging, config-discovery shape, and where clone logic lives. Page-vs-integrate gate: N/A — no app UI or nav is touched; docs additions are content pages in the existing Starlight site.

Option 1: Shared-drizzle-spine remote backend + core copy() + bin-in-package CLI

recommended

Extract the drizzle query layer from backend/cloudflare.ts into a shared module generic over an async drizzle SQLite database. The remote backend feeds it a drizzle-orm/sqlite-proxy instance whose callback POSTs to D1's HTTP /raw endpoint; R2 data goes through a DataStorage adapter over Bun.S3Client. copy(source, dest) lives in library core with restic-style hash-skip incremental semantics. The CLI ships as a bin inside @f0rbit/corpus (source under cli/, compiled into dist/cli/) with zero new runtime deps. Config discovery: optional corpus.config.ts (drizzle-kit pattern, needed for codecs) + wrangler-file sniffing + env vars for connection params.

Real-world examples

  • drizzle-kit / drizzle-orm — the d1-http driver takes {accountId, databaseId, token} for laptop→D1 control-plane access, and runtime HTTP queries go through sqlite-proxy's callback shape; chosen so tooling needs only an API token, no Worker. source, source
  • wrangler — d1 execute --remote POSTs to the same D1 REST query endpoint; the CLI deliberately reuses the dashboard REST control-plane so a laptop needs only an account API token. source
  • restic — restic copy transfers snapshots incrementally, moving only blobs missing at the destination, implemented against one narrow Backend interface shared by all storage providers. source
  • DVC — push/pull diffs local cache against remote CAS by content hash; sync is a storage-layer capability, not a UI feature. source

Pros

  • One drizzle query layer serves both the Workers-binding backend and the HTTP backend — schema changes in schema.ts flow to both; they cannot drift.
  • The remote backend is a real Backend, so it plugs into run_backend_contract_tests unchanged; every CLI command is backend-generic for free.
  • Zero new runtime dependencies: node:util parseArgs, Bun.S3Client, native fetch, and drizzle-orm is already a dependency — bin-in-package costs the library nothing.
  • Single module graph: bunx corpus resolves the bin from the consumer's own @f0rbit/corpus install — the exact same package instance the consumer's corpus.config.ts imports. No dual-package hazard for codecs, brands, or Result helpers.
  • copy() in core is reusable by other consumers (e.g. pipelines server-side seeding) and property-testable with the 0.7.0 testing substrate.

Cons

  • Refactoring backend/cloudflare.ts onto a shared layer is the riskiest single step — mitigated by the existing contract suite running the real backend against the bun:sqlite D1 fake.
  • Typing the shared layer generically over drizzle's BaseSQLiteDatabase variants (d1 vs sqlite-proxy) takes care; drizzle's generics are deep.
  • The published package carries a bun-only bin (Bun.S3Client) although the library itself stays runtime-agnostic — acceptable: the bin only executes when invoked, and the CLI's audience runs bun.

Architecture decisions

Non-obvious calls within Option 1. None require user sign-off; all are documented here as the record.

  • Read-only is CLI policy, not a backend property. The remote backend implements the full MetadataStorage/DataStorage surface (D1 HTTP supports INSERT/DELETE; S3 supports PUT). This keeps it pluggable into run_backend_contract_tests unchanged. The v1 CLI simply exposes no mutating commands; clone writes only to its destination backend. Docs recommend read-scoped tokens (D1 read + R2 Object Read Only) as the enforcement layer that actually matters.
  • No apply_batch on the remote backend in v1. Backend.apply_batch is optional, and every transaction-contract test self-skips via if (!backend.apply_batch) return — verified against tests/integration/backend-contract.test.ts:560-660. The D1 HTTP endpoint's multi-statement semantics don't guarantee a single transaction, so claiming atomicity would be a lie. corpus.transaction() falls back to sequential + compensating deletes if anyone runs one against remote.
  • list_stores as an optional MetadataClient method. The CLI's stores command and un-filtered copy() need store enumeration, which MetadataClient lacks. Adding it as optional (list_stores?) is non-breaking for custom backends. Implementations: memory = distinct store_id over the meta Map; file = existing list_all_stores() walker (keeps the _*/.* filter); drizzle shared layer = selectDistinct on store_id; layered = delegate to the bottom (write) layer. copy() and the CLI return err({ kind: "invalid_config", ... }) with an actionable message when the method is absent and no explicit --store filter was given.
  • Copy ordering: data before metadata. A version "exists" at the destination iff its metadata row exists — the same metadata-is-source-of-truth reasoning as the cloudflare apply_batch orphan policy. Interrupted clones leave orphan data objects at the destination, which are harmless (content-addressed) and healed/skipped on re-run. Skip rules: version skipped when dest.metadata.get(store, version) is ok; data transfer skipped when dest.data.exists(meta.data_key) — so clone twice is a no-op second time.
  • copy() preserves SnapshotMeta verbatim — version, created_at, content_hash, data_key, parents, tags all copied untouched (a clone is a replica, not a re-ingest). Consequence: custom data_key_fn layouts land at the destination with source keys, which is where the file-backend data_path sanitisation aliasing gotcha (backend/file.ts:64, /→_) can bite. Documented in the clone docs; NOT fixed here (tracked in .plans/corpus-testing-substrate.html out-of-scope).
  • Bun.S3Client over aws4fetch. Zero-dep and native; the CLI/remote backend runs exclusively under bun on laptops/CI, never in Workers. Wrapped behind the DataStorage interface so tests drive fakes. Read path needs only GetObject/HeadObject (data keys come from metadata rows — no ListObjects needed). Public types of backend/remote-r2.ts are structural (mirroring how backend/cloudflare.ts types D1/R2 structurally) so emitted .d.ts never references the Bun global.
  • D1 transport: sqlite-proxy against the /raw endpoint. drizzle(async (sql, params, method) => ...) where the callback POSTs to POST /accounts/{account_id}/d1/database/{database_id}/raw — rows-as-arrays matches the proxy driver's {rows} contract directly. The response envelope is Zod-parsed (d1_http_response_schema) before rows reach drizzle; HTTP/auth failures map to storage_error with the operation name.
  • define_config lives in a pure module. cli/config-schema.ts holds the CorpusCliConfig type, its Zod schema (for the data-shaped parts; store definitions/codecs are validated structurally since functions aren't Zod-able), and the define_config identity helper — no bun/node APIs, so the main barrel re-exports it and Workers consumers can import types safely. The loader (cli/load-config.ts: upward search for corpus.config.ts, dynamic import — bun imports TS natively) is bun-only and never enters the barrel.
  • Connection resolution precedence: explicit flags > corpus.config.ts environments > wrangler sniff + env vars. Secrets (API token, R2 keys) come ONLY from env vars or flags — never from the config file. Env vars: CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID, CORPUS_R2_ACCESS_KEY_ID, CORPUS_R2_SECRET_ACCESS_KEY (R2 S3 creds derive from an R2 API token: key id = token id, secret = SHA-256 of token value — documented, not computed).
  • Wrangler sniffing scope: wrangler.toml via Bun's native TOML support, wrangler.json/wrangler.jsonc via a minimal comment/trailing-comma-stripping pass + JSON.parse, both Zod-validated down to the three facts we need: account_id, d1_databases[].database_id, r2_buckets[].bucket_name. Ambiguity (multiple D1 bindings) is an error naming the candidates, resolvable via config or flags.
  • CLI structure: bin: { "corpus": "./dist/cli/index.js" } with a #!/usr/bin/env bun shebang (tsc preserves shebangs). Two-stage node:util parseArgs: first positional selects the command, remainder re-parsed against that command's option spec. All printing funnels through cli/output.ts (injectable sink for tests, --json mode) — the ONLY file with a no-console exception, added to the config exception maps with a why-comment (precedent: testing/registry.ts).
  • Command surface (v1): corpus stores, corpus versions <store> [--tag ... --limit --before --after], corpus show <store> <version> [--observations], corpus cat <store> <version> [--raw], corpus lineage <store> <version> [--depth], corpus clone <source> <dest> [--store ... --tag ... --dry-run --concurrency]. Backend selectors: --env <name> (config environment; implicit remote when wrangler + creds suffice) or --file <path>; clone source/dest accept an env name or a filesystem path. v1 clone destinations must be file backends.
  • Versioning: lands in the still-unpublished 0.8.0 line (0.7.0 testing substrate is also unpublished). Additive only — whether to publish 0.7.0 and 0.8.0 separately or together is the user's call and out of this plan.
