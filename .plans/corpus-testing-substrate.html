Plan: corpus testing substrate

todo

Ship a property-based testing substrate inside @f0rbit/corpus exposed as a single testing namespace. Hybrid deriver (hand-rolled walker over @traversable/zod-test as leaf), Hedgehog-style cover assertions, error-path-exhaustive runtime laws, model-based provider_equivalence, and a package.json entry-point vending protocol so downstream packages (vault, pulse, devpad-pipelines) contribute arbitraries that compose automatically. Phase 1 pilots the substrate against corpus's own backend-contract suite — the densest cross-backend equivalence target in the ecosystem.

Root cause

Stated goal: Build a property-based testing substrate inside @f0rbit/corpus under a testing namespace.

Underlying need: ~30 projects share one Result<T, CorpusError>-shaped vocabulary but each codebase reinvents PBT: bespoke arbitraries, ad-hoc generators per error branch, no shared model-based test for the in-memory-vs-real Provider pattern that the design philosophy mandates. The cost is silent under-coverage of error paths (rate limits, transaction aborts, partial commits) and divergence between in-memory fakes and real backends — the exact bugs PBT is supposed to surface. What the user actually wants is a substrate that composes across the dependency graph so leaf packages stay small, error paths are proven to fire (not hoped to), and Provider equivalence is a one-liner.

Why this matters: Solving the surface request as a thin fc.Arbitrary re-export buys nothing — the ecosystem already has fast-check. The leverage is in the registry-as-composition-mechanism (downstream packages' arbitraries flow up through the dependency graph), cover assertions that fail when labelled branches don't fire, and error_path_exhaustive driven by the Result error union — these three together convert "we hope our tests cover the unhappy paths" into a compile-time-checked, runtime-verified guarantee.

Alternatives considered

Option 1: Hybrid deriver + hybrid registry key + fast-check shrinking + walk-on-init vending

recommended

Hand-rolled Zod walker for the common cases (object, array, union, branded, refinement) that delegates to @traversable/zod-test for esoteric leaves we don't want to maintain (z.bigint, z.date refinements, complex catch). Registry keyed by branded type symbols for identity-bearing domain types (UserId, AuthToken, Money, CorpusError variants) and by Zod schema instance identity (WeakMap-keyed) for everything else. Shrinking inherits from fast-check's native rose-tree — we wrap only the generator step in compose + failure, never the shrink path. Vending walks node_modules once on first testing import, reading package.json's "corpus": { "testing": "./dist/testing/register.js" } key, dynamic-importing each, executing in undefined order (registry is order-independent).

Real-world examples

  • Effect Schema — Arbitrary.make(schema) ships inside the schema package (not a separate deriver), single source of truth, schema = source for type + parse + arb. source
  • Hypothesis (Python) — st.register_type_strategy + entry-points in pyproject.toml via the "hypothesis" key let downstream packages contribute arbitraries that registration-order-independently compose. We translate setuptools entry-points to package.json keys. source
  • fast-check 4.7 — native rose-tree shrinking, fc.commands for model-based testing. Wrapping the shrink layer is documented as fragile; the release notes explicitly call out that derivers should stay at the generator layer. source
  • @traversable/zod-test — only credible Zod v4 + fast-check 4 deriver currently maintained; we use it as the leaf primitive for the awkward cases we don't want to track upstream changes for. source
  • Hedgehog — integrated shrinking + cover assertions inside Gen; the precedent for "fail the test if a labelled branch fires <N%". We borrow the assertion, not the integrated-shrinker architecture. source
  • Proptest (Rust) — #[derive(Arbitrary)] + prop_compose!. Our testing.compose((draw) => ...) is the JS analog; derivation falls out of the Zod walker. source

Pros

  • Single import surface (testing.*) and single source of truth (Zod schema or branded symbol — never duplicated).
  • Hand-rolled walker covers ~85% of cases corpus + downstream actually use (objects, unions, arrays, primitives, refinements); leaves the long tail to a maintained leaf. Avoids the zod-fast-check rot pattern (whole-system dependency on one unmaintained deriver).
  • Branded keys give compile-time error if you register two generators for the same domain type. Schema-identity keys let unbranded ad-hoc schemas register without ceremony.
  • Walk-on-init is order-independent; mirrors Hypothesis's proven model; no codegen step, no scaffolder coupling.
  • Inheriting fast-check shrinking means compose + failure wrappers stay ~50 LOC each; we never touch the rose-tree.

Cons

  • Hand-rolled walker is real code to maintain — ~600 LOC across object/array/union/primitive/branded handlers. Mitigated by delegating leaves to @traversable/zod-test and by Phase 1's pilot exercising the walker against corpus's own schemas (high signal).
  • Two registry-key paths (branded vs schema-identity) is conceptual surface area — testers must know which to use. Mitigated by the rule "if you have a branded type, key on the brand; otherwise key on the schema".
  • Walk-on-init pays a one-time node_modules traversal cost on first testing import. Bounded by reading only top-level package.jsons + the declared paths; in practice <50 ms even at the 30-package ecosystem scale.

Architecture decisions

Non-obvious calls within the chosen approach. None require user weigh-in — flagged here as audit trail.

  • Subpath, not subpackage. Corpus is a single-package library (per AGENTS.md). Testing lives at testing/ at the repo root, exposed via a new ./testing entry in package.json exports. Consumers import as import { testing } from "@f0rbit/corpus" for the namespace, or import * as testing from "@f0rbit/corpus/testing" if they want only the testing surface without pulling the rest of corpus into a hot import path. The namespace re-export from the main barrel preserves the user's "single import surface" requirement.
  • fast-check is a dependency, not peerDependencies. Corpus is the only place fast-check is configured. Downstream packages don't import fast-check directly — they go through testing.*. This means corpus pins the fast-check major version for the entire ecosystem. @traversable/zod-test is also a regular dependency for the same reason.
  • Registry is a module-level singleton with a reset hook for tests. testing.__reset_registry_for_tests() is exported but namespaced under __ to signal "do not use in normal tests". Required because the registry's walk-on-init side effect is observable; without a reset hook, registry behaviour tests are flaky.
  • Vending walker reads only top-level package.jsons of declared dependencies (not transitive). Order: walk process.cwd()/package.json's direct deps (incl. devDeps), each resolved via require.resolve's package root, read each package.json, follow the corpus.testing key. Transitive registration happens because each package's registrar can register on behalf of its own deps. This avoids the O(N²) hoisted-monorepo walk.
  • The cover assertion uses fast-check's built-in fc.statistics mechanism rather than re-implementing labelled-branch counting. We wrap fc.assert with a post-run hook that compares observed proportions against the declared minimum and throws if any label is below threshold. ~150 LOC, no shrink-path involvement.
  • error_path_exhaustive is type-driven, not value-driven. The function under test must return Promise<Result<T, E>> where E is a tagged union ({ kind: "..." } | ...). The law extracts the set of kind values via a mapped type at the call site, looks up each in the registry, and asserts each registered generator produces the matching variant. Missing variant = TS error at call site. Unregistered variant at runtime = test failure with a clear message ("no failure generator registered for kind 'rate_limited' — call testing.failure(YourError, 'rate_limited', ...) in your test setup or registrar").
  • Provider equivalence model is bring-your-own. We don't ship a generic model class — the substrate accepts a model object that implements the same interface as the SUT, plus an array of fc.AsyncCommands. Phase 1's pilot defines the canonical corpus example (model = memory backend's Maps, SUT = each real backend); downstream packages copy that pattern.
  • Auto-load is idempotent and lazy. First access to any testing.* export triggers the walker once; subsequent accesses skip. Tests that need to override registration order can call testing.__reset_registry_for_tests() + manual testing.load_from(...).
  • Phase 5 scaffolder integration is documentation + a single template file — we do not modify devpad pipelines init from the corpus repo. The plan ships a testing/register.ts template under docs/templates/ that the devpad scaffolder copies. Cross-repo coupling stays one-way (devpad reads from corpus templates, not the reverse).

Context (self-contained)

The registry stores fc.Arbitrary<T> instances keyed two ways:

  1. Branded symbol — for identity-bearing domain types. The brand is a unique symbol declared by the consumer, paired at the type level with a TS branded type (type UserId = string & { readonly __brand: unique symbol }). Implementation uses Map<symbol, fc.Arbitrary<unknown>>.
  2. Zod schema instance — for ad-hoc schemas. Keyed via WeakMap<z.ZodType, fc.Arbitrary<unknown>> so schemas can be garbage-collected and identity comparison is referential.

The public API surface for this task is:

// testing/types.ts
export type ArbBrand<T> = symbol & { readonly __arb_type?: (x: T) => void }
export type FailureKey<E extends { kind: string }> = E["kind"]

// testing/registry.ts
export function arbitrary<T>(key: ArbBrand<T>, gen: fc.Arbitrary<T>): void
export function arbitrary<S extends z.ZodType>(schema: S, gen: fc.Arbitrary<z.infer<S>>): void
export function lookup<T>(key: ArbBrand<T>): fc.Arbitrary<T> | undefined
export function lookup<S extends z.ZodType>(schema: S): fc.Arbitrary<z.infer<S>> | undefined
export function failure<E extends { kind: string }, K extends E["kind"]>(
  brand: ArbBrand<E>,
  variant: K,
  gen: fc.Arbitrary<Extract<E, { kind: K }>>
): void
export function lookup_failure<E extends { kind: string }>(
  brand: ArbBrand<E>,
  variant: E["kind"]
): fc.Arbitrary<E> | undefined
export function __reset_registry_for_tests(): void

Failure registrations live in a separate Map<symbol, Map<string, fc.Arbitrary<unknown>>> indexed by [error_brand, variant_kind]. The phantom __arb_type property on ArbBrand<T> is purely structural — it carries T at the type level without runtime overhead.

Result types come from ./types.ts at repo root (NOT from @f0rbit/corpus — see AGENTS.md rule about importing within this repo).

Context (self-contained)

The walker translates a Zod schema into an fc.Arbitrary. It first checks the registry (schema-identity lookup) — if a generator is registered for this schema, return it. Otherwise dispatch on Zod's _def.typeName (ZodObject, ZodArray, ZodUnion, ZodLiteral, etc.) to produce arbitraries for the common cases. Esoteric leaves (ZodBigInt with refinements, ZodDate ranges, ZodCatch, ZodPipe, ZodLazy with cycles) delegate to @traversable/zod-test's generator(schema).

Signature:

export function arb<S extends z.ZodType>(schema: S): fc.Arbitrary<z.infer<S>>

Cases the hand-rolled walker handles directly:

  • ZodString: fc.string(), honour .min/.max/.uuid/.email/.regex via fast-check's fc.stringMatching + filters.
  • ZodNumber: fc.float/fc.integer based on .int(), honour .min/.max.
  • ZodBoolean: fc.boolean().
  • ZodLiteral: fc.constant(literal_value).
  • ZodEnum/ZodNativeEnum: fc.constantFrom(...values).
  • ZodObject: fc.record({ key: arb(child), ... }), recurse.
  • ZodArray: fc.array(arb(element), { minLength, maxLength }).
  • ZodTuple: fc.tuple(arb(each)).
  • ZodUnion/ZodDiscriminatedUnion: fc.oneof(arb(each)).
  • ZodOptional/ZodNullable: fc.option(arb(inner)) with appropriate nil.
  • ZodBranded: arb(unwrap) — brand is type-level only.
  • ZodDefault: arb(inner) (we generate the input shape; the default never fires).
  • ZodRecord: fc.dictionary(key_arb, arb(value)).
