# corpus

A functional, versioned-snapshot library for TypeScript. Provides typed stores with content-hash deduplication, lineage tracking, and observations (typed annotations attached to snapshot pointers). Ships multiple backends: in-memory, file, Cloudflare D1+R2, and a layered cache.

## Layout

This is a **single-package** library (not a monorepo). All source lives at the repo root.

Typecheck, tests, and lint are all required CI gates — `.github/workflows/ci.yml`'s `test` job runs typecheck → `bun test` → `bun run lint && bun run fmt:check` → build, and the `test` status check is required by the `protect-main` branch ruleset (no merge without it passing). Lint exceptions live as `files`-scoped overrides — `eslint.config.ts` for the typed ESLint layer, `.oxlintrc.json` for the oxlint layer (never inline `eslint-disable`): `concurrency.ts` (Semaphore class), `corpus.ts`/`result.ts`/`testing/**` (documented intentional throws; `result.ts` also hosts the sanctioned try/catch → Result boundary), `tests/**` (functional rules + `must-use-result` relaxed). `.oxfmtrc.json` must stay a byte-copy of the canonical `@f0rbit/oxfmt-config` copy (`f0rbit-lint check` enforces); repo-specific format ignores go in `.prettierignore` — which excludes ALL of `tests/fixtures/` because fixture trees are byte-exact walker test data and one `package.json` is deliberately malformed. `bunfig.toml` pins `linker = "hoisted"` (the umbrella's transitive bins need it — don't remove).

PRs on this repo merge with **squash only** — merge commits and rebase merges are disabled (settings flipped 2026-07-04; history before 0.7.0 predates this). The PR title becomes the squash commit title, so PR titles follow git-workflow commit rules: no "phase"/"task"/plan identifiers, describe the change itself.

One-time bootstrap after cloning: `git config blame.ignoreRevsFile .git-blame-ignore-revs` — keeps the oxfmt normalization commit out of `git blame`.

## Conventions

These differ from or override global defaults — read them.

### Error handling — Result type

This library defines its **own** `Result<T, E = CorpusError>` and `ok` / `err` helpers in `types.ts`. **Do not import `@f0rbit/corpus` from inside this repo** for Result types; import from `./types.js` (or `./result.js` for combinators). The global `corpus-patterns` skill describes the same API — its examples apply, just with relative imports.

Conventions:

- Never throw across module boundaries. Use `try_catch` / `try_catch_async` to wrap external calls (fs, fetch, Drizzle, R2) at the edge.
- Use `pipe()` chains for multi-step Result transformations rather than nested `if (!result.ok) return result`.

### Style

- Naming (snake_case variables/functions incl. exports, PascalCase types, kebab-case filenames) is machine-enforced — `naming: "snake_case"` in `eslint.config.ts`'s `define_lint_config` call plus oxlint's `unicorn/filename-case: kebabCase`. See "Linting & formatting" below for the exception map when a name can't comply (documented public-API exceptions live there, not here). Source files are flat at repo root (`utils.ts`, `corpus.ts`), not nested into kebab-case directories.
- Functional, composition-first. No classes except `Semaphore` (intentional — it owns mutable internal state; config-scoped exception, see below).
- Indentation is oxfmt-owned (tabs, width 120, config in `.oxfmtrc.json` sourced from `@f0rbit/oxfmt-config`) — run `bun run fmt`; never hand-align or hand-pick tabs vs. spaces file-by-file.
- **No JSDoc churn** — existing source carries rich `@example` blocks because typedoc/llms.txt extract from them. When you add an exported function, mirror the surrounding doc style. (Not machine-enforced — stays a review convention.)

### Linting & formatting

Toolchain is the `@f0rbit/lint` umbrella (exact-pinned in `devDependencies` at `0.3.0`; sub-packages `@f0rbit/eslint-config`, `@f0rbit/eslint-plugin`, `@f0rbit/oxfmt-config`, `@f0rbit/oxlint-config` resolve at the same version through the umbrella's own `dependencies` — no `overrides` block needed). Two layers:

- **oxlint** — fast, syntactic. Runs first (`bun run lint` = `oxlint . && eslint .`); fails in milliseconds so it's the first thing CI reports.
- **typed ESLint** — `typescript-eslint` `strictTypeChecked` + `eslint-plugin-functional` (no-classes, no-throw, no-try discipline) + the custom `f0rbit/*` org rules, `@typescript-eslint/consistent-type-definitions: "type"`, and a full `no-console` ban. `eslint-plugin-oxlint` de-dupes the overlap so no rule fires twice.

Org rules active as of 0.3.0 (tier map: error unless noted):

- **`f0rbit/must-use-result`** — every `Result<T, E>` returned from a function is actually checked (`.ok`, `match`, `pipe`, etc.) rather than silently dropped.
- **`f0rbit/require-schema-at-boundary`** — raw external data (`JSON.parse`, fetch-`Body`-like `.json()`) must flow through a Zod parse before use; `as`-casting a raw parse result is flagged. Fixed corpus's on-disk/DB-row boundary at `utils.ts` (`parent_ref_schema` + `raw_snapshot_meta_schema`, consumed by `parse_snapshot_meta` and `backend/file.ts`'s `_meta.json` read). Two scoped overrides remain (see exception map) — `result.ts`'s generic `fetch_result` default and `observations/storage.ts`'s round-trip of corpus's own `derived_from` column.
- **`f0rbit/no-test-mocks`** — no `mock`/`spyOn`/`jest` from `bun:test`; use in-memory fakes. Fixed 11 hits across `tests/unit/{result,testing-autoload,testing-failure,testing-registry}.test.ts` — `result.test.ts`'s stray `mock()` wrapper became a plain function (it already reassigns `globalThis.fetch`, same pattern as its existing `mock_fetch` helper); the three `spyOn(console, "warn")` assertions moved to a new shared fake, `tests/fakes/console.ts` (`fake_console_warn()` — reassigns `console.warn` and records calls, mirroring `mock_fetch`'s style).
- **`f0rbit/no-ambient-effects`** (warn) — `Date.now()`/`new Date()`/`Math.random()` outside `ambient_effect_files`. No file in corpus qualifies as a dedicated clock/rng provider module (the reads sit inside general-purpose files — `utils.ts`'s `generate_version`, `observations/utils.ts`'s `generate_observation_id`, `corpus.ts`'s put paths, `observations/storage.ts`'s `create_observation_row`) — designating any of them via `ambient_effect_files` would over-suppress the rest of that file, so these 32 warns (6 production + 26 in test fixtures) are left as graduated warns (see README `## todo`) rather than churning the API.
- **`f0rbit/prefer-pipe`** (warn) — 2+ consecutive manual Result unwraps. Fixed 3 mechanical single-value cases in `corpus.ts` (`get`, `get_latest`, `delete`, now `pipe(...).flat_map(...).result()`). 7 remain, all inside `corpus.transaction(async (tx) => ...)` test bodies that thread 2-3 prior Result values into a later step — `pipe()`'s single-threaded chain doesn't compose cleanly for that shape; graduated to README `## todo` rather than nesting pipes for dubious readability gain.
- **`@typescript-eslint/consistent-type-definitions: "type"`** — no-op for corpus; the codebase already uses `type` exclusively (zero `interface` declarations pre-existing).
- **`no-console`** (full ban) — two scoped overrides remain (see exception map); no other hits.

Exception maps — new intentional exceptions go in one of these two, each entry with a why-comment; **never** an inline `eslint-disable`/oxlint disable comment (`reportUnusedDisableDirectives` is on, so an undocumented inline disable is itself a lint failure):

- `eslint.config.ts` — `overrides: [{ files: [...], rules: {...} }]` array passed to `define_lint_config`. Current entries: `concurrency.ts` (Semaphore + its try/finally), `backend/file.ts` (`apply_batch`'s structural try/catch), `corpus.ts` (`build()`'s intentional throw), `result.ts` (the sanctioned throw/try-catch → Result boundary, plus `f0rbit/require-schema-at-boundary` off for `fetch_result`'s generic `parse_body` default), `testing/**` (construction throws) and a couple of narrower per-file casts/naming allowances (`testing/arb.ts`, `testing/cover.ts`, `types.ts`/`corpus.ts`, `observations/**` — now including `observations/storage.ts`'s `f0rbit/require-schema-at-boundary` off for its self-written `derived_from` column, `sst.ts`, `version-set.ts`, `testing/registry.ts`, `testing/vending/auto-load.ts`), `testing/registry.ts` + `testing/vending/**` (`no-console` off — documented degrade-to-warn UX), `tests/fakes/console.ts` (`no-console` off — it IS the console fake), `tests/**` (functional + `must-use-result` relaxed for test code).
- `.oxlintrc.json` — `overrides` array, same shape. Current entries: `tests/**` (non-null assertions after explicit setup), `testing/**` and `version-set.ts` (documented `_`/`__`-prefixed dunder allowances), `scripts/generate-llm-docs.js` (`__dirname` shim), `types.ts`/`corpus.ts` (`Store<any>` variance escape).

### Testing

- `bun test` only. No vitest, no jest.
- Backend integration tests share a single contract suite at `tests/integration/backend-contract.test.ts`. New backends should plug into it via `run_backend_contract_tests(name, factory, cleanup)`.
- Real I/O over mocks: file backend tests touch `tests/.tmp/` (cleaned per-test).
- The Cloudflare backend runs through the contract suite against in-memory **platform fakes** at `tests/fakes/cloudflare.ts` — a `bun:sqlite`-backed D1 fake and a Map-backed R2 fake. The code under test is the real `backend/cloudflare.ts` (real Drizzle d1 driver, real SQL). Real-platform smoke via `wrangler dev` remains out-of-band; the fakes are not a substitute for a deploy check.
- Per the global `testing-strategy` skill: in-memory fakes, integration-first, Provider pattern. Memory backend itself doubles as the in-memory fake for downstream consumers.

### Schema / migrations

Drizzle is the sole source of truth: `schema.ts` (`corpus_snapshots`) and `observations/schema.ts` (`corpus_observations`). The snapshots table is exposed via `@f0rbit/corpus/schema`; `corpus_observations` is re-exported through the main barrel (`observations/index.ts`). Corpus ships **no** migration SQL and no `drizzle-kit` history — consumers point `drizzle-kit generate` at the exported tables and own their migration history (see `docs/src/content/docs/guides/cloudflare.mdx`). `sst.ts` is only the `createCorpusInfra` resource-naming helper; the former `CORPUS_MIGRATION_SQL` constant was removed in commit `6aba89c`.

### Version sets

The `version_set_store` factory lives in `version-set.ts`. The manifest schema (`VersionSetManifest`) is consumed by `~/dev/devpad/packages/pipelines/` and any package scaffolded by `devpad pipelines init`. Treat any change to the Zod schema as a breaking change across consumers.

- _Zod `.default()` makes input ≠ output_: prefer a bidirectional `extends` assertion over `satisfies ZodType<T>` when the static type mirrors a schema with defaults. `.default()` widens the input type but narrows the output, so a direct `satisfies` against the inferred shape rejects schemas with defaults.
- _Per-store partitioning via internal tag_: stores that need a per-partition `data_key_fn` (e.g. `version_set_store`) read the partition key from a tag the store sets on every `put`. Keep this tag set as a private convention of the store factory — consumers shouldn't have to know to set it themselves.

### Observations

Observations are versioned annotations on snapshot pointers (store + version + optional JSONPath + span). They live in a separate Drizzle table and a separate client (`corpus.observations`). They're optional — only enabled when `observations` is wired into the backend. Internally, `observations/` is treated as its own subsystem with its own types and adapter interface — keep observation logic out of `backend/*.ts`.

### Testing substrate

Property-based testing primitives live in `testing/`, exported under the `@f0rbit/corpus/testing` entry point and as a namespace re-export (`import { testing } from "@f0rbit/corpus"`). Prefer the direct entry point in hot paths — it skips the main barrel's transitive imports.

- **Two import paths, one surface**: `import * as testing from "@f0rbit/corpus/testing"` OR `import { testing } from "@f0rbit/corpus"`. Both resolve to `testing/index.ts`, which exposes `arb`, `compose` (+ `Draw` type), `arbitrary`, `lookup`, `failure`, `lookup_failure`, `list_registered_variants`, `load_from`, `cover` (+ `CoverageError`, `CoverageLabel`, `CoverageStat` types), the command surface (`commands`, `async_commands`, `model_run`, `async_model_run` + `Command`/`AsyncCommand`/`ICommand`/`CommandsConstraints` types), the `law` namespace (`law.error_path_exhaustive`, `law.round_trip`, `law.idempotent`, `law.functor`, `law.provider_equivalence`, `law.equivalence_command`), `register` + the brand symbols (`CORPUS_ERROR_BRAND`, `SNAPSHOT_META_BRAND`, `BATCH_OP_BRAND`), `__reset_registry_for_tests`, `ArbBrand`, `FailureKey`, and `fc` (the fast-check default export re-exported as a convenience so consumers don't pull `fast-check` separately). Corpus is the only place `fast-check` is configured; downstream packages MUST go through `testing.*`, not import `"fast-check"` directly.
- **Hybrid registry — two key types**: branded `unique symbol` keys for identity-bearing domain types (UserId, AuthToken, Money, error brands like `CORPUS_ERROR_BRAND`), and `WeakMap<z.ZodType, ...>` keyed by Zod schema instance for ad-hoc schemas without a brand. WeakMap is deliberate — schemas can be GC'd; a plain Map would leak. Duplicate registration overwrites with a `console.warn`. `__reset_registry_for_tests()` clears all three maps (brand, schema, failure).
- **`testing.arb(schema)` is hand-roll only in Phase 1.** The plan originally wired a fallback to `@traversable/zod-test`'s `generator(schema)` for esoteric Zod leaves; that package declares `zod@4` as a peer (corpus is `zod@3`) and its `@traversable/zod-types` peer is missing in the install graph, so the fallback was never functional. `@traversable/zod-test` remains in `dependencies` (hook left open for later) but is NOT imported in `testing/arb.ts`. Do not "fix" the missing import — revisit only when corpus moves to zod 4. Unsupported schema kinds throw with an actionable message pointing at the manual escape hatch: `testing.arbitrary(schema, gen)`.
- **Walker uses `instanceof`, not `_def.typeName`.** Each branch narrows the schema type natively — no `as` cast per branch. The single `as fc.Arbitrary<z.infer<S>>` at the registry boundary is the only escape-hatch cast in `testing/arb.ts`. Adversaries should flag any other.
- **Zod v3 internals worth knowing** (encountered during walker implementation): `z.ZodDiscriminatedUnion` is NOT a `z.ZodUnion` — check the discriminated case FIRST or it falls through wrong. `ZodObject._def.shape` is a thunk — call `_def.shape()`, don't read the property. `ZodLazy` recursion needs `fc.letrec` PLUS `fc.oneof({ depthSize: "small", maxDepth: 3 })` to actually bound — `fc.letrec` alone doesn't.
- **fast-check 4.x gotchas:** `fc.hexaString` was removed — use `fc.stringMatching(/^[a-f0-9]{N}$/)`. `fc.emailAddress()` produces RFC-5322 addresses that Zod's `.email()` rejects (Zod is stricter); supply a tighter custom arbitrary when generating against `z.string().email()`. `fc.date()` includes BC-era timestamps that Zod's `.datetime()` regex rejects; clamp to a sensible window. The composite generator API is `fc.gen()` (not `fc.context`) — `testing.compose` is a thin wrapper over it. **`fc.gen()`'s shrinker does not reliably terminate** (observed unbounded on fast-check 4.8 — a single-draw compose arbitrary with an always-failing property hangs `fc.assert` for 90s+ while the equivalent `fc.record` shrinks in ms). Any `fc.assert` over a `compose`-based arbitrary where the property can fail MUST disable shrinking via `{ endOnFailure: true }`; `testing.law.error_path_exhaustive` does this internally. `fc.sample` is unaffected.
- **Property tests live under `tests/property/`** (new sibling to `tests/unit/` and `tests/integration/`). `bun test` picks them up automatically; the pilot at `tests/property/backend-roundtrip.test.ts` runs 200 generated `SnapshotMeta` values through the memory backend's metadata round-trip.
- **SnapshotMeta arbitrary coverage gap (intentional, not a bug)**: the `SNAPSHOT_META_BRAND` arbitrary always emits `parents: []` and never emits `invoked_at`. Still not widened as of phase 3 — the equivalence pilot uses a richer local `meta_arb` (parents, `invoked_at`, `tags`) instead of the registered brand arbitrary. Don't tighten downstream tests against the current registered shape.
- **`CORPUS_ERROR_BRAND` is the canonical brand for `CorpusError`.** All 12 variants have failure generators registered by `register()` in `testing/register.ts`. Downstream packages register additional failure cases against this imported symbol — never mint their own brand for `CorpusError`. Adding a 13th `CorpusError` variant: TypeScript breaks every `error_path_exhaustive` provoke table until the variant gets an entry, and `register()` needs a matching `failure(CORPUS_ERROR_BRAND, ...)` generator.
- **`testing.law.error_path_exhaustive` recipe**: type the `provoke` table over the FULL error union (`{ [K in E["kind"]]: (failure) => Args }` — compiler enforces coverage), then scope runtime execution with `only: [...]` to the variants the function actually emits; give non-surfaceable variants throwing stubs so the table stays honest. Reference implementation: `tests/property/corpus-error-paths.test.ts`. Note: no production code path emits `hash_mismatch` today — the pilot's wrapper adds content-hash verification on top of `store.get_handle`.
- **`testing.cover` labels are checked AFTER the property succeeds** — if the property itself fails, fast-check's counterexample fires and the coverage report is suppressed. Intentional (the bug comes first). Shortfalls throw `CoverageError` whose message lists ALL labels (met flagged `OK`, unmet `FAIL`) and which carries structured `stats` for programmatic use.
- **Vending protocol (active since phase 4)**: a package contributes arbitraries by declaring `"corpus": { "testing": "./path/to/register.js" }` in its `package.json`. The registrar exposes an explicit `register()` — it does NOT side-effect on import, so consumers can import brand symbols for identity without registering anything. (Walker contract: `load_registrars` calls `register()` when exported; a module WITHOUT one is still loaded and assumed to register via import side-effects — supported fallback, not the convention.) Registration is order-independent and contributes via `testing.arbitrary(...)` + `testing.failure(...)`. The walker (`testing/vending/walker.ts`) reads `{cwd}/package.json`, includes the cwd package itself, and inspects DIRECT deps only (dependencies + devDependencies + peerDependencies) — no transitive walk; a direct dep re-registers its own deps' arbitraries in its registrar if it wants them vended. Hoisted monorepos work because resolution goes through `import.meta.resolve` (with a `{cwd}/node_modules/...` fallback for packages whose `exports` map omits `"./package.json"`).
- **Auto-load semantics**: the walker fires on the first `testing.lookup` or `testing.lookup_failure` call, at most once per registry state; concurrent first callers share one walk. `testing.arb(schema)` and `testing.compose` do NOT trigger auto-load — `arb()` consults the internal sync accessors (`lookup_sync`/`lookup_failure_sync`) because deriving an arbitrary is a synchronous API and must not spring filesystem walks. Consequence: `lookup`/`lookup_failure` are **async** (await them); the sync signature never shipped in any published release (introduced and made async within unreleased 0.7.0). Auto-load never throws — discovery/load failures degrade to a `console.warn` pointing at `testing.load_from(...)`.
- **Within-repo override + escape hatch**: `testing.__reset_registry_for_tests()` clears the three registry maps AND the auto-load promise (next lookup re-walks). `testing.load_from(package_name_or_dir)` explicitly loads ONE package's registrar — the escape hatch for packages auto-discovery can't see (non-installed fixture trees, monorepo siblings) or for deterministic control in tests. It's additive: it doesn't mark auto-load as done. The `__` prefix flags non-production use.
- **Dev-checkout guard (why corpus's own suite still imports its registrar manually)**: `package.json`'s self-registrar key points at `./dist/testing/register.js`. In the corpus dev checkout that's a parallel module graph with its OWN registry instance — loading it would double-register invisibly and make `bun test` depend on `bun run build`. The walker therefore skips the cwd package when it is named `@f0rbit/corpus`, and corpus's suite imports `testing/register.ts` from source (`backend-roundtrip`, `corpus-error-paths`, `testing-failure` tests). The auto-load contract: consumers get corpus + their declared deps automatically; the corpus dev checkout registers itself from source. Do NOT "fix" the guard or de-wire the manual imports.
- **Fixture `node_modules` trees are intentionally git-tracked** via `.gitignore` negation (`!tests/fixtures/**/node_modules/`) — the walker tests need real on-disk dependency layouts. If you add a fixture tree, verify `git ls-files | grep node_modules` still lists only intended fixture files.
- **Model-based equivalence — `testing.law.provider_equivalence` recipe**: build commands with `testing.law.equivalence_command({ label, on_model, on_sut })` — both arms share the result type `R` so they can't drift at the type level. Per-command results ride a module-private `WeakMap` mailbox keyed by the model instance (the only value fast-check's command wrapper passes through untouched); commands themselves are stateless, so instances are safe to reuse across sequence positions, property runs, and shrink retries. The law builds a FRESH model + FRESH SUT per provider per run (runtime-enforced) — `model` / provider `build` factories MUST NOT capture state across runs. Pass `size: "max"` when deep stateful histories matter: fast-check's default sizing averages ~5 commands per sequence even with `maxCommands: 30`. Payload arbitraries inside commands must be plain `fc.*`, not `testing.compose` (the `fc.gen()` shrink hazard above — sequence shrinking is deliberately left ON in this law). Canonical reference implementation: `tests/property/backend-equivalence.test.ts` (model + commands + per-step equivalence) — cross-implementation equivalence tests in downstream packages should copy its structure.
- **Equivalence-rule convention** (for `provider_equivalence` callers): `ok` flags must agree; on ok, structural-equal modulo Date precision (compare epoch ms) and list ordering (canonical-sort both arms — stronger than a set, still order-independent); on err, only `error.kind` must match — messages and causes legitimately differ across backends.
- **`fc.asyncCommands` does not exist in fast-check 4.8** — `fc.commands` is overloaded over sync/async command arrays; corpus pins the async overload as `testing.async_commands` so an upstream reshape becomes a compile error here, not a runtime failure downstream. Related: on `fc.assert` failure, fast-check 4.x attaches the property's thrown error as `error.cause`; the top-level message only carries the counterexample — assert against `(e as Error).cause` in tests.
- **Per-law run-count defaults are module-local**, tuned per cost profile (`error_path_exhaustive` 50/variant, `round_trip`/`idempotent` 100, `functor`/`cover` 200). Deliberately NOT exported from the laws barrel — pass `numRuns` to override. Only `testing.DEFAULT_NUM_RUNS` (cover's 200) is public.
- **Starlight docs reference**: long-form guides and examples live in `docs/src/content/docs/testing/` — the Testing sidebar group in the Starlight site. AGENTS.md captures the quick reference; consult Starlight for tutorials, worked examples, and decision flowcharts.
- **Template stub for new packages**: `docs/templates/testing/register.ts` is a fully-commented template that downstream packages (scaffolded via `devpad pipelines init`) copy when wiring up testing arbitraries. It demonstrates brand registration, Zod schema usage, and the `register()` export pattern.

## Gotchas

- `create_corpus().build()` **throws** when no backend is configured — the one intentional throw in the library. It's a config-time programmer-error guard on builder misuse (unreachable from runtime data); converting `build()` to return a Result would break the public API. Everything downstream of `build()` returns `Result`.
- `CorpusError` is a **discriminated union** — switch on `error.kind`, not `error.message`. Adding a new kind is a breaking change to consumers (TypeScript will force exhaustiveness).
- The `Corpus` type carries `create_pointer`, `resolve_pointer`, `is_superseded` as methods on the corpus instance, not just exports. Both code paths exist; the instance methods are the documented API.
- `define_store` accepts an optional `data_key_fn(ctx)` to override the default `${store_id}/${content_hash}` data key. Custom layouts (e.g. partition by tag/date) go through this — don't reimplement key computation upstream.
- Cloudflare backend uses `eq()` (not `like()`) for `store_id` filtering in `metadata.list()`. Don't regress this — see commit `02bee7f`.
- `MetadataClient.list_stores` (`types.ts`) is **optional** (`list_stores?: () => AsyncIterable<string>`) — added non-breaking so custom backends aren't forced to implement enumeration. All built-in backends implement it (memory: `Set` over map values; file: delegates to the pre-existing `list_all_stores()` walker, preserving its `_*`/`.*` filter; drizzle shared layer: `selectDistinct`; layered: delegates to the bottom write layer), all sorted for determinism. Consumers that need enumeration (the CLI, `copy()`) must handle absence with an actionable error rather than treating it as empty. Contract-suite coverage self-skips per backend via `if (!backend.metadata.list_stores) return` — same pattern as the `apply_batch` transaction guards.
- `peerDependencies` lists `zod ^3` and `typescript ^5`. Codecs that take a Zod schema should accept the schema generically (`z.ZodType<T>`), not pin to a specific Zod version's class.
- Cross-store transactions on the Cloudflare backend can leave R2 orphan blobs when a transaction aborts after data writes but before D1 commit. This is intentional (data is content-addressed and idempotent; metadata is the source of truth). A follow-up `corpus.gc()` is tracked in the README.
- `Codec<T>` has optional `encode_stream` / `decode_stream`. `text_codec` and `binary_codec` ship both; `json_codec` ships neither (Zod needs the full document). `compose(head, ...layers)` includes stream methods only when every layer has them — structural inference, no `StreamableCodec<T>` brand.
- `compose()` decode pipes streams through layers natively. `compose()` encode buffers between each layer (the `encode_stream` signature is `(value: T) => ReadableStream<Uint8Array>`, not stream-to-stream — this is intentional for head codecs but buffers when `T = Uint8Array` for layers). Documented in `.plans/streamed-reads.md` §2.3 as a Phase 4 follow-up if it becomes a real bottleneck.
- `Bun.file().stream()` emits 256 KB chunks. Streaming-correctness tests need a payload ≥ 512 KB to reliably assert `>= 2` chunks; smaller payloads give exactly one chunk and look identical to a buffered read.
- `DataStorage.get` returns a `DataStorageHandle` (with `bytes()` and optional `stream()`), not raw bytes — since 0.4.0. Custom backends built on `create_data_client` migrate via `wrap_bytes_storage(...)` from `backend/base.ts`. The consumer-facing `DataHandle` (in `types.ts`) is unchanged.
- `corpus.transaction()` read-your-writes covers `tx.get` against in-tx puts AND in-tx deletes via tombstones. The handle tracks `buffered_meta` (in-tx puts) AND `tombstoned_meta` (in-tx deletes); both must stay in sync if you add a new mutating op to the handle. Without the tombstone, `tx.get` after `tx.delete` of a pre-existing committed version would fall through to the live backend and incorrectly return the soon-to-be-deleted value.
- Memory backend's `apply_batch` snapshots its closure-local Maps shallowly. Set `CORPUS_DEV=1` to `Object.freeze` stored `SnapshotMeta` / `ObservationRow` and catch consumer-side mutation bugs that would corrupt the rollback snapshot. Off by default — `Object.freeze` is hot-path expensive.
- File backend `apply_batch` stages writes under `<base>/.tx-<uuid>/` and atomic-renames into the live tree on commit. `list_all_stores()` filters out both `_*` (internal: `_data`, `_observations.json`) and `.*` (staging dirs) — when adding directory-walking code anywhere in the file backend, mirror this filter.
- `recover(root_dir)` is exported from `@f0rbit/corpus/file` (not the main barrel). Startup-time helper that removes leftover `.tx-*` staging dirs from a previous crash. Do NOT invoke concurrently with a live process pointing at the same dir — there's no file locking.
- File-backend transactions are per-store-metadata atomic but the batch of renames across files is NOT atomic: a crash mid-commit can land some renames and not others, surfaced as `partial_commit`. Recovery semantics are abort-only — no roll-forward.
- The fake D1 in `tests/fakes/cloudflare.ts` implements exactly the surface drizzle-orm's d1 driver calls: `prepare(sql)` → `bind(...params)` → `{ run(), all() → {results}, raw() → rows-as-arrays }`, plus atomic `batch(statements)` (drizzle's `SQLiteD1Session.batch` binds each statement then calls `client.batch`). Its DDL is derived from the Drizzle tables via `getTableConfig` — schema stays the single source of truth, and unsupported schema features (defaults, FKs, partial indexes, SQL expression indexes) throw loudly so the fake can't silently diverge. If a drizzle upgrade starts calling a new D1 method, the contract suite fails fast on the missing method.
- Layered-backend `apply_batch` forwards to the bottom write layer only. Transactional writes skip cache layers; the cache fills lazily on subsequent reads or eagerly when the consumer issues non-transactional writes (which fan out to all layers). Mixing tx and non-tx writes against a layered backend produces asymmetric cache state — drive all writes through one path if cache consistency matters.
- File-backend data-key aliasing: `data_path` (`backend/file.ts:64`) sanitises `/` → `_` when mapping a data key to a file name, so keys `a/b` and `a_b` collide on the same `.bin` file. The default `${store_id}/${content_hash}` layout never collides (single `/`, hex hash — no `_`), but custom `data_key_fn` layouts can. Surfaced by the phase-3 equivalence pilot (which constrains its key domain to avoid it); fix tracked in `.plans/corpus-testing-substrate.html` `#out-of-scope`.
- `parse_snapshot_meta`'s parameter type (`utils.ts`) is `z.infer<typeof raw_snapshot_meta_schema>`, not a hand-written object literal — the Zod schema (plus its sibling `parent_ref_schema`) is the single source of truth for the raw on-disk (`_meta.json`) / D1-row shape, validated via `.parse(...)` at `backend/file.ts`'s `_meta.json` read and inside `parse_snapshot_meta` itself (the `parents`/`tags` JSON-string branches). Added during the `@f0rbit/lint` 0.3.0 `require-schema-at-boundary` burn-down — if you touch the raw meta shape, update the schema, not a duplicated type literal.
- `bun run lint` scans the whole tree via `oxlint . && eslint .`; neither `.oxlintrc.json` nor the eslint-config factory's `global_ignores` excludes `.claude/` (it's `.gitignore`d, not lint-ignored). A live agent worktree under `.claude/worktrees/<id>/` — created by the harness's `EnterWorktree` tool and left mounted while that agent session is active — is a full checkout and gets scanned too, doubling every diagnostic and polluting per-rule counts. Not fixed here (touching shared lint config to work around another concurrent session felt out of scope); if it recurs, filter survey output by path or add `.claude/**` to both ignore configs.
